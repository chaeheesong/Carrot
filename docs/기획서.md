# 당근마켓 오마주 (GraphQL 웹 서비스) 기획서

## 0. 프로젝트 목표
- **핵심 목표**: GraphQL의 강점(오버/언더페칭 해결, 실시간 Subscription, 타입 안전성)을 눈에 보이는 형태로 시연
- **범위**: 홈 피드 / 상품 상세 / 실시간 채팅, 3개 기능만 빠르게 구현
- **플랫폼**: 웹 (반응형)
- **최종 목표**: 실제 배포된 URL로 접속 가능한 상태

---

## 1. 기술 스택

| 영역 | 선택 | 이유 |
|---|---|---|
| 프론트엔드 | **Next.js 14 (App Router) + TypeScript** | 웹 배포가 Vercel로 매우 빠름, React 생태계, SSR로 초기 로딩도 빠름 |
| GraphQL 클라이언트 | **Apollo Client** | 캐싱, Subscription 지원이 가장 성숙함 |
| 스타일 | **TailwindCSS** | 빠른 UI 작업, 당근마켓 느낌의 카드 UI 구현에 적합 |
| 백엔드 | **Node.js + Apollo Server + TypeScript** | 프론트와 언어 통일 → 타입 공유 가능, 학습/개발 속도 ↑ |
| ORM | **Prisma** | 스키마 기반 타입 자동생성, GraphQL 스키마와 궁합 좋음 |
| DB | **PostgreSQL (Supabase)** | 무료 티어로 빠르게 시작, 이미지 저장(Storage)도 같이 해결 |
| 실시간 | **GraphQL Subscription (graphql-ws)** | 채팅 기능에 GraphQL 고유 강점을 그대로 적용 |
| 인증 | **JWT 기반 자체 인증** (또는 Supabase Auth 활용) | 빠르게 붙일 수 있고 GraphQL 리졸버에서 컨텍스트로 검증하기 쉬움 |
| 배포 (프론트) | **Vercel** | Next.js 공식 지원, Git push만 하면 자동 배포 |
| 배포 (백엔드) | **Railway** | Vercel 서버리스는 WebSocket(Subscription) 유지가 어려워서, 상시 구동 서버가 필요한 Railway/Render 사용 |

**왜 백엔드/프론트를 분리하나요?**
Next.js API Routes 안에 GraphQL을 넣을 수도 있지만, Subscription(WebSocket)은 서버리스 환경과 궁합이 나빠서 별도의 Node 서버(Railway)로 분리하는 게 안정적입니다.

## 2. 전체 아키텍처

```
[Next.js Web (Vercel)]
        │  HTTPS (Query/Mutation)
        │  WSS (Subscription)
        ▼
[Apollo Server on Railway] ── Prisma ── [PostgreSQL (Supabase)]
        │
        └── [Supabase Storage] (상품 이미지)
```

- 일반 조회/등록: HTTP POST (Query/Mutation)
- 실시간 채팅 메시지: WebSocket (Subscription)
- 이미지 업로드: 클라이언트 → Supabase Storage 직접 업로드 → URL만 GraphQL Mutation으로 저장

## 4. 기능 명세 (해야 할 일 단위로)

### 4.1 홈 피드
**목적**: 동네 기준으로 판매중인 상품을 카드 리스트로 보여준다.

- [ ] 로그인 시 저장된 동네(neighborhood) 기준으로 상품 목록 조회
- [ ] 상품 카드에 표시: 썸네일, 제목, 가격, 등록시간(n분 전), 찜 개수, 판매자 닉네임
- [ ] 무한 스크롤 (cursor 기반 페이지네이션)
- [ ] 카드 클릭 시 상품 상세로 이동
- [ ] 찜(좋아요) 토글 버튼 — 목록에서 바로 클릭 가능
- [ ] (선택) 카테고리 필터 드롭다운

**GraphQL 쿼리 예시**
```graphql
query HomeFeed($neighborhood: String!, $cursor: ID) {
  products(neighborhood: $neighborhood, after: $cursor, first: 20) {
    edges {
      node {
        id title price thumbnailUrl createdAt
        likeCount isLikedByMe
        seller { nickname }
      }
    }
    pageInfo { hasNextPage endCursor }
  }
}
```

### 4.2 상품 상세
**목적**: 상품 하나를 볼 때 필요한 모든 연관 정보를 한 번의 요청으로 가져온다.

- [ ] 상품 이미지(여러 장), 제목, 설명, 가격, 상태(판매중/거래완료)
- [ ] 판매자 정보 + 판매자가 등록한 다른 상품 목록(최대 3~5개)
- [ ] 문의 댓글 목록 + 댓글 작성
- [ ] "채팅하기" 버튼 → 기존 채팅방 있으면 이동, 없으면 새로 생성
- [ ] 찜하기 버튼

**GraphQL 쿼리 예시**
```graphql
query ProductDetail($id: ID!) {
  product(id: $id) {
    title description price status
    images { url }
    seller {
      nickname
      otherProducts(first: 3) { id title thumbnailUrl }
    }
    comments { content author { nickname } createdAt }
  }
}
```

**Mutation**
```graphql
mutation StartChat($productId: ID!) {
  createOrGetChatRoom(productId: $productId) { id }
}
```

### 4.3 실시간 채팅
**목적**: 구매자-판매자가 상품 기준으로 1:1 대화를 실시간으로 주고받는다.

- [ ] 채팅방 목록: 상대방 닉네임, 마지막 메시지, 관련 상품 썸네일, 안 읽은 메시지 수
- [ ] 채팅방 진입 시 과거 메시지 히스토리 로드 (cursor 페이지네이션)
- [ ] 메시지 전송 (Mutation)
- [ ] 새 메시지 실시간 수신 (Subscription) — 상대방이 보낸 메시지가 새로고침 없이 도착
- [ ] (선택) 읽음 표시

**GraphQL 예시**
```graphql
query MyChatRooms {
  myChatRooms {
    id
    partner { nickname }
    product { title thumbnailUrl }
    lastMessage { content createdAt }
    unreadCount
  }
}

subscription OnNewMessage($chatRoomId: ID!) {
  messageAdded(chatRoomId: $chatRoomId) {
    id content createdAt
    sender { nickname }
  }
}

mutation SendMessage($chatRoomId: ID!, $content: String!) {
  sendMessage(chatRoomId: $chatRoomId, content: $content) { id }
}
```

## 5. 인증 (최소 범위)
- 이메일 + 비밀번호 로그인/회원가입 (소셜 로그인은 오마주 범위에서 제외 — 필요시 2단계에서 추가)
- 회원가입 시 동네는 텍스트 목록에서 선택 (GPS 인증은 이번 범위에서 제외)
- JWT를 Apollo Server 컨텍스트에서 검증 → 로그인 필요한 리졸버에서 체크

## 6. 개발 순서 (마일스톤)

| 단계 | 내용 |
|---|---|
| 1단계 | Prisma 스키마 설계, DB 세팅(Supabase), 인증 붙이기 |
| 2단계 | 홈 피드 (Query + 프론트 카드 UI) |
| 3단계 | 상품 상세 (Query + Mutation, 이미지 업로드 연동) |
| 4단계 | 실시간 채팅 (Subscription 서버/클라이언트 연동) |
| 5단계 | Vercel + Railway 배포, 환경변수 정리, 최종 QA |

---

## 7. 배포 계획
1. **DB/Storage**: Supabase 프로젝트 생성 → Postgres 연결 문자열 확보, Storage 버킷 생성
2. **백엔드**: Railway에 GitHub 연동 → Apollo Server 배포 → 환경변수(DB URL, JWT secret) 설정
3. **프론트**: Vercel에 GitHub 연동 → Next.js 배포 → 환경변수(GraphQL API URL, WS URL) 설정
4. **도메인**: Vercel 기본 도메인(예: `xxx.vercel.app`)으로 우선 배포, 커스텀 도메인은 선택사항

## 8. 아직 정해지지 않은 것 (진행하면서 여쭤볼 부분)
- 카테고리(가전/의류 등) 시스템을 넣을지, 아니면 카테고리 없이 갈지
- 거래 완료 처리(상태값)까지 구현할지, 아니면 등록/조회/채팅까지만 할지
- 로그인 없이 게스트로 피드는 볼 수 있게 할지 (읽기는 공개, 액션은 로그인 필요 식으로)
